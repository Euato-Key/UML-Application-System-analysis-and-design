这是一个非常典型的软件工程实验，任务量适中但链路较长。为了保证效率，建议采用**“角色分工+流水线协作”**的模式。

我们将四位同学（吴俊安、帅军锐、吴思扬、林世颖）分为三个技术角色和一个全栈/集成角色。

以下是具体的任务分配建议：

### 👥 角色分配概览

| 姓名 | 角色定位 | 核心职责 (关键词) |
| :--- | :--- | :--- |
| **吴俊安** | **PM & 需求分析师** | 访谈、Prompt工程、用例图、活动图、文档整合 |
| **帅军锐** | **系统架构师** | 类图设计、序列图、PlantUML精修、设计一致性把控 |
| **吴思扬** | **知识图谱工程师** | Neo4j环境、运行脚本(build/trace)、生成追踪链、图谱可视化 |
| **林世颖** | **开发工程师** | 生成代码(Java/Python)、本地运行、单元测试、变更代码实现 |

---

### 📋 详细任务清单

#### 1. 吴俊安（主攻：阶段1 & 阶段2 & 报告统筹）
你是项目的起点，负责把“模糊的需求”变成“清晰的文档”。
*   **模拟访谈：** 准备一段关于“智能候诊系统”的模拟对话文本（可以直接用实验指导书的例子或稍微扩充）。
*   **Prompt 编写：** 利用 Kimi 将文本转为 **业务术语表** 和 **用例图**。
*   **流程建模：** 针对核心用例，让 AI 生成 **活动图**（Activity Diagram），并进行人工校验，确保逻辑通顺。
*   **交付物：** `1_biz` 和 `2_req` 文件夹内容。
*   **额外任务：** 负责最终实验报告的汇总和排版，确保文档规范。

#### 2. 帅军锐（主攻：阶段3 - 设计）
你是项目的骨架，负责把“文字需求”变成“技术图纸”。
*   **类图设计：** 根据吴俊安提供的用例，让 AI 生成 **类图**。**注意：** 必须按要求在类图中加入 `需求追踪：对应UC编号` 的备注。
*   **序列图设计：** 选取一个核心场景（如扫码签到），生成详细的 **序列图**。
*   **UML 审查：** AI 生成的 PlantUML 代码经常有小错误，你需要负责在 VS Code 中调试这些图，确保能渲染出正确的图片。
*   **交付物：** `3_design` 文件夹（含所有 .puml 文件）。

#### 3. 林世颖（主攻：阶段4 & 阶段5 - 代码与变更）
你是项目的肌肉，负责把“图纸”变成“可运行的代码”。
*   **代码生成：** 拿到帅军锐的类图后，投喂给 Kimi，要求生成 Spring Boot (Java) 或 Python 代码。**关键点：** 必须要求 AI 在代码注释里加上 `@trace UCxx` 标记。
*   **环境跑通：** 在本地建立工程，确保代码能编译通过，接口文档（Swagger）能打开。
*   **变更实现：** 针对阶段5的“预约签到”变更，生成新的 SQL 和 增量代码，并跑通单元测试。
*   **交付物：** `4_code` 和 `5_change` 文件夹。

#### 4. 吴思扬（主攻：工具链串联 & 知识图谱）
你是项目的神经系统，负责“全链路追踪”。
*   **环境搭建：** 负责搞定 Neo4j AuraDB（云端）或本地 Neo4j 的配置，解决端口冲突等疑难杂症。
*   **脚本运行：**
    *   拿到帅军锐的 `.puml` 文件，运行 `kg_build.py` 生成设计图谱。
    *   拿到林世颖的 `代码目录`，运行 `kg_trace.py` 生成代码追踪关系。
*   **变更分析：** 在阶段5，执行 Cypher 查询语句，截图“影响分析”的结果（高亮受影响的节点）。
*   **交付物：** Neo4j 截图、`kg_dump` 数据导出文件。

---

### 🔄 协作流水线（按时间顺序）

1.  **[T+0] 吴俊安** 完成需求文档和用例图 -> **传给 帅军锐**。
2.  **[T+1] 帅军锐** 完成类图和序列图 (.puml文件) -> **分发给 林世颖 和 吴思扬**。
    *   *注意：此时类图必须定稿，类名不能再随意改，否则代码对不上。*
3.  **[T+2] 并行工作：**
    *   **林世颖** 根据类图生成并调试代码。
    *   **吴思扬** 利用 .puml 文件生成初步知识图谱。
4.  **[T+3] 追踪集成：**
    *   **林世颖** 将调试好的代码包 发给 **吴思扬**。
    *   **吴思扬** 运行追踪脚本，扫描代码注释，完善知识图谱。
5.  **[T+4] 变更演练：**
    *   小组开会讨论“预约签到”变更。
    *   **吴思扬** 在图谱中做影响分析截图。
    *   **林世颖** 用 AI 生成变更代码。
6.  **[T+5] 吴俊安** 收集所有人的截图和文档，撰写最终报告，大家互评签字。

### ⚠️ 给小组长的特别提示
*   **脚本哪里来？** 实验指导书里提到了 `kg_build.py` 和 `kg_trace.py`。如果没有这两个文件，吴思扬需要第一时间找老师/助教获取，或者去实验指定的 GitHub 仓库下载。
*   **保持沟通：** 最容易出问题的地方是 **“类图里的类名”** 和 **“代码里的类名”** 不一致。帅军锐和林世颖需要密切配合，确保命名规范统一。
